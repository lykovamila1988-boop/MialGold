2. DATA FLOW — FROM USER TO AGENT RESPONSE
============================================

SCENARIO 1: Interactive Chat (Browser)
──────────────────────────────────────

USER (Browser)
  │
  ├─→ types message in "Marina" tab
  │
  └─→ POST /api/agent/marina/message
      {
        "message": "Напиши 3 поста про тревогу",
        "session_id": "abc123",
        "chain_id": "optional_existing_chain"
      }
      │
      ├─→ session_manager.get_history("marina", session_id)
      │   → returns [{"role": "user", "content": "..."}]
      │
      ├─→ system_prompt_builder.extract_context_from_message()
      │   → finds [from:agent] tags if present
      │
      ├─→ base.compose_system("marina", MARINA_SYSTEM, context)
      │   → adds phase info + prompt overrides + brand voice
      │
      ├─→ base.run_agent(client, system, tools, handle, msg, history)
      │   │
      │   ├─→ client.messages.create(system=system, messages=...)
      │   │
      │   ├─→ Model sees:
      │   │   • System prompt (with context if present)
      │   │   • Available tools (JSON schemas)
      │   │   • Message history
      │   │
      │   ├─→ Model responds: "Вот 3 поста..." (with tool calls)
      │   │
      │   ├─→ for tool_call in response:
      │   │   ├─→ marina.handle(tool_name, args)
      │   │   │   • read_file("02-content/...")
      │   │   │   • write_file(...)
      │   │   │   • run_command("python tools/...")
      │   │   │
      │   │   └─→ append tool result to messages
      │   │
      │   └─→ repeat until model stops calling tools
      │
      ├─→ memory.log_event("agent:message", {...})
      │   → appends to events.jsonl
      │
      ├─→ session_manager.save_history("marina", session_id, history)
      │   → saves updated chat history
      │
      └─→ return {"reply": "Вот посты...", "context": {...}}
          │
          └─→ Browser displays reply + next actions buttons


SCENARIO 2: Chain from n8n Workflow
────────────────────────────────────

n8n (Remote)
  │
  ├─→ HTTP POST http://127.0.0.1:5052/api/n8n/trigger-chain
      {
        "chain_config": {
          "agents": ["olya", "marina", "victoria", "vasya"]
        },
        "from_agent": "n8n",
        "chain_id": "content_week_2026_06_08_001"
      }
      │
      ├─→ n8n_webhook.py receives & validates
      │   ├─→ checks Authorization header
      │   ├─→ creates ChainExecution object
      │   └─→ spawns subprocess: python pipeline.py content_week
      │
      ├─→ pipeline.py runs chain non-interactively:
      │   │
      │   ├─→ for position, agent_key in enumerate(agents):
      │   │   │
      │   │   ├─→ build context:
      │   │   │   {
      │   │   │     "from_agent": agents[position-1] or "system",
      │   │   │     "chain_id": "content_week_2026_06_08_001",
      │   │   │     "position": position,
      │   │   │     "is_final": position == len(agents)-1
      │   │   │   }
      │   │   │
      │   │   ├─→ base.run_agent(client, system, tools, handle, msg, history)
      │   │   │   (same as scenario 1)
      │   │   │
      │   │   ├─→ save_checkpoint(chain_name, {"completed": [agents[:pos]]})
      │   │   │   → allows resume on failure
      │   │   │
      │   │   └─→ memory.log_event("chain:step", {agent, status, duration})
      │   │
      │   └─→ memory.log_event("chain:end", {status: "ok|failed"})
      │
      ├─→ n8n_webhook.py detects completion
      │   │
      │   └─→ POST to N8N_STATUS_WEBHOOK_URL
          {
            "ok": true,
            "chain_id": "content_week_2026_06_08_001",
            "status": "success",
            "result": "5 posts scheduled..."
          }
          │
          └─→ n8n continues next steps (email, Telegram, etc)

7. FILE OPERATIONS & PATHS
===========================

When agent calls: base.read_file("02-content/content-plan.md")

Resolution:
  1. Input path: "02-content/content-plan.md"
  2. _apply_alias(path):
     "02-content" → "MILA-BUSINESS/02-content" (from _FOLDER_ALIASES)
  3. Full path: "MILA-BUSINESS/02-content/content-plan.md"
  4. _safe_path():
     Resolve within MILA_FOLDER (E:\MILA GOLD)
     Check: resolved path is under MILA_FOLDER
     → E:\MILA GOLD\MILA-BUSINESS\02-content\content-plan.md
  5. Read file (max 40KB into context)
  6. If larger than 40KB:
     Return: [first 40KB] + "…[file truncated, request specific section]"

Write operation: base.write_file("02-content/posts/new.md", content)
  1. Input: "02-content/posts/new.md"
  2. Apply aliases: "02-content" → "MILA-BUSINESS/02-content"
  3. Safe path: E:\MILA GOLD\MILA-BUSINESS\02-content\posts\new.md
  4. Create parent dirs if missing: mkdir -p
  5. Write to temp file: .tmp
  6. Atomic rename: .tmp → final
  7. Return: "✓ Сохранено: [full path]"

Security: Path traversal prevention
  ✗ Input: "../../../etc/passwd"
  → Resolves to: E:\MILA GOLD\..\..\etc\passwd
  → Check fails: not under MILA_FOLDER
  → Raise ValueError("Path outside workspace")

  ✓ Input: "02-content/../../../etc/passwd"
  → Resolves to: E:\etc\passwd
  → Check fails: not under MILA_FOLDER
  → Raise ValueError("Path outside workspace")


8. MEMORY SHARING & LOCKING
============================

Memory files (under mila-office/memory/):

┌─────────────────────────────────────────────────────────────┐
│ SHARED STATE (inter-process via file locking)               │
└─────────────────────────────────────────────────────────────┘

context.json
  Latest input from n8n or user
  {
    "kind": "new_client",
    "from": "telegram",
    "data": {"name": "Маша", ...},
    "timestamp": "2026-06-08T10:22:45Z"
  }

  When agent starts: reads context.json to understand trigger
  When n8n sends new event: overwrites context.json (atomic)

  Locking: _FileLock prevents concurrent writes

profile.json
  Office profile (stable, not per-event)
  {
    "business": {
      "brand": "@liudmyla.lykova",
      "phase_override": null,
      "ig_followers": 1300,
      ...
    },
    "positioning": {...}
  }

  When agent starts: reads profile → sets defaults
  Updated manually or by Manager agent
  NOT overwritten per event (unlike context.json)

events.jsonl
  Append-only audit log (one event per line)
  [{"kind": "chain:start", ...}, {"kind": "chain:step", ...}, ...]

  Agents append to this via memory.log_event()
  Never overwritten, only appended
  Dashboard reads this for timeline/metrics

agent_notes.json
  Agent-to-agent message queue
  {
    "notes": [
      {
        "from": "marina",
        "to": "victoria",
        "message": "Check post grammar",
        "timestamp": "..."
      }
    ]
  }

  Marina writes: agent_notes.append({from: "marina", to: "victoria", ...})
  Victoria reads: filter where to="victoria"
  After processing: mark as read

Locking mechanism (synchronous):
  ┌─ base.py (run_agent, tools)
  │  │
  │  └─→ needs to write to memory
  │     │
  │     ├─→ acquire _FileLock(timeout=15s)
  │     │   └─→ create file: .lock
  │     │   └─→ if exists, exponential backoff
  │     │   └─→ if timeout: assume dead process, force
  │     │
  │     ├─→ read JSON (atomic)
  │     ├─→ modify
  │     ├─→ write to temp file
  │     ├─→ atomic rename (temp → final)
  │     │
  │     └─→ release _FileLock
  │        └─→ delete .lock file
  │
  └─ Next agent can now acquire lock


9. ERROR HANDLING FLOW
======================

Exception raised anywhere (agent, tool, API call)
  │
  ├─→ error_monitor.log_error(e, context={...}, alert=bool)
  │   │
  │   ├─→ Categorize error:
  │   │   • config_error: Missing .env var (ANTHROPIC_API_KEY)
  │   │   • policy_violation: Agent not allowed (policies.py)
  │   │   • rate_limit: HTTP 429 (retry helps)
  │   │   • timeout: 504, network timeout (retry helps)
  │   │   • agent_error: Agent output invalid (escalate)
  │   │   • task_failed: Agent can't complete (review)
  │   │
  │   ├─→ Write to logs/errors.jsonl:
  │   │   {
  │   │     "timestamp": "2026-06-08T10:45:23Z",
  │   │     "error_type": "instagram_api.GraphError",
  │   │     "error_message": "429 Too Many Requests",
  │   │     "context": {
  │   │       "agent": "marina",
  │   │       "tool": "run_command",
  │   │       "command": "python tools/post_content.py ..."
  │   │     },
  │   │     "traceback": "..."
  │   │   }
  │   │
  │   └─→ If alert=True:
  │       ├─→ POST to Telegram (TELEGRAM_ADMIN_CHAT_ID)
  │       │   "⚠️ MILA Office Error\n{error_type}: {message}\nAgent: {agent}"
  │       │
  │       └─→ Log: "error_id" = UUID for reference
  │
  └─→ Decide: RETRY or ESCALATE or FAIL?
      │
      ├─→ If config_error: FAIL (no retry)
      ├─→ If rate_limit: RETRY (exponential backoff)
      ├─→ If agent_error: ESCALATE (different agent)
      ├─→ If task_failed: LOG & ALERT (manual review)
      └─→ If transient network: RETRY (up to 3 times)
